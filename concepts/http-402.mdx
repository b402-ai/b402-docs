[HTTP 402](https://datatracker.ietf.org/doc/html/rfc7231#section-6.5.2) is a standard HTTP status code meaning "Payment Required." It was reserved for decades but rarely implemented. b402 activates it as a programmable payment layer for agents across EVM-compatible chains.

## How It Works in b402

When a client requests a paid resource:

1. The server responds with `402 Payment Required`
2. The response body contains structured payment instructions (amount, token, recipient, relayer contract)
3. The client constructs and signs an EIP-712 payment payload
4. The client resends the request with the payment in the `X-PAYMENT` header
5. The server verifies and settles the payment via the facilitator

## Why HTTP 402?

**Native to the web.** No new transport protocols, sidechains, or payment channels. Standard HTTP with JSON payloads.

**Stateless.** Each request carries its own payment proof. No sessions, no accounts, no API keys for buyers.

**Machine-readable.** AI agents can detect `402` responses, parse payment requirements, and pay autonomously.

**Composable.** Any HTTP client (curl, fetch, axios, AI agent frameworks) can integrate b402 payments by handling one status code.

## b402 vs x402

b402 extends the [x402 protocol](https://x402.org) with additional capabilities:

| Feature | x402 | b402 |
|---------|------|------|
| Tokens | USDC only | USDT, USDC, USD1 |
| Privacy | None | RAILGUN integration |
| Wallets | EOA only | EOA + Smart Wallets (ERC-7579) |
| Gas | Buyer pays | Gasless (facilitator covers gas) |
| Swaps | None | PancakeSwap V3 integration |
| Identity | None | ERC-8004 agent identity |

The payment flow is compatible with x402 at the HTTP level but adds capabilities at the service and protocol layers.
